# 武汉东湖学院毕业论文生成总流程清单

本清单用于指导 agent 从项目资料生成或重写武汉东湖学院本科毕业论文初稿。主指南解决 Word 格式问题，封面规范解决前两页问题；本清单解决“如何从任务书、开题报告、源码、截图和图表参考中组织论文内容”的问题。

## 1. 执行入口

开始任何论文生成或大范围修改前，按以下顺序读取：

1. 项目根目录 `AGENTS.md`：确认本项目要求使用 Superpowers plugin skills。
2. `武汉东湖学院本科毕业论文Word格式与排版规则指南.md`：确认 Word 格式、结构和最终验收规则。
3. `封面前两页可执行规范.md`：确认中文封面和英文封面生成方式。
4. `目录页可执行规范.md`：确认目录页 TOC 样式、制表位、点线前导符和标题层级要求。
5. `自动化脚本原理与复用规范.md`：确认 Word COM、OpenXML、封面复制、目录更新和验证脚本的复用方式。
6. 本文件：确认内容分析、章节组织和证据约束流程。
7. 学校原始文件：`论文格式.doc`、`论文初稿写法及要求.docx`、`范本.docx`、`任务书.docx`、`开题报告.docx`。
8. 项目材料：源码目录、系统功能截图目录、论文各种图表的样式参考目录。

如果入口文件之间有冲突，优先级为：学校正式要求 > 范本版面 > 本指南可执行规则 > agent 默认经验。

## 2. 输入资料盘点

生成前必须建立输入资料清单，至少覆盖以下对象：

| 资料 | 必须提取的信息 | 用途 |
|---|---|---|
| `任务书.docx` | 题目、学生、专业、指导教师、任务目标、进度要求 | 封面字段、摘要、绪论、任务边界 |
| `开题报告.docx` | 研究背景、研究意义、技术路线、预期功能、进度安排 | 绪论、需求分析、设计目标 |
| `论文格式.doc` | 学校正式格式要求 | 最高格式依据 |
| `论文初稿写法及要求.docx` | 初稿阶段补充要求 | 初稿内容取舍和风险控制 |
| `范本.docx` | 封面、声明、摘要、目录、正文、参考文献版面 | 模板级复制和视觉核对 |
| 源码目录 | 技术栈、目录结构、数据模型、接口、核心功能、AI/算法实现 | 需求分析、系统设计、系统实现、测试 |
| 系统功能截图 | 页面功能、交互流程、测试结果、界面证据 | 系统实现、系统测试、图示材料 |
| 图表样式参考 | 流程图、模块图、E-R 图、表格风格 | 图表重绘和格式核对 |

若某项资料缺失，不能自行补造事实；应在修改说明中标注“未提供/未能验证”，并用已验证材料组织论文。

## 3. 证据台账

写正文前必须建立证据台账。台账不一定交付给用户，但 agent 必须用它约束内容。

| 证据编号 | 来源文件或目录 | 已验证事实 | 可写入章节 | 禁止扩写 |
|---|---|---|---|---|
| E1 | 任务书 | 论文题目、任务目标 | 封面、摘要、绪论 | 不得改题目 |
| E2 | 开题报告 | 研究背景、技术路线 | 绪论、需求分析 | 不得照搬成大段重复文本 |
| E3 | 源码配置文件 | 技术栈、依赖、运行方式 | 开发环境、系统实现 | 不得写不存在的框架 |
| E4 | 数据库/模型文件 | 数据表、字段、关系 | 数据库设计、E-R 图 | 不得编造字段 |
| E5 | 路由/接口/控制器 | 功能模块、接口流程 | 功能需求、实现章节 | 不得写未实现功能 |
| E6 | AI/算法相关代码 | 模型输入、输出、预测逻辑 | AI 模型设计、实现 | 不得夸大算法复杂度 |
| E7 | 截图 | 页面和测试结果 | 实现章、测试章 | 不得把截图外功能写成已实现 |

正文中的每个核心结论、功能描述、表结构、测试结果都应能回溯到至少一个证据来源。

## 4. 源码分析流程

分析源码时按以下顺序，不要先写论文正文：

1. 查看项目根目录和 README、配置文件、依赖文件，确认技术栈和运行方式。
2. 识别前端、后端、数据库、脚本、模型或算法模块的目录边界。
3. 查找入口文件、路由文件、控制器、服务层、数据访问层、模型文件。
4. 提取用户角色、功能模块和核心业务流程。
5. 提取数据表、字段、主键、外键或逻辑关联，形成数据库设计依据。
6. 提取 AI 功能的输入数据、处理过程、输出结果、调用位置和局限性。
7. 查找测试、日志、示例数据或截图证据，确认能写入测试章节的内容。
8. 记录无法确认的功能，不写成已实现功能。
9. 形成项目专用验证清单，至少列出后端 API、前端视图、模型/数据表、AI 特征与回退逻辑、总 E-R 图必须出现的主题。

常用查找方向：

| 目标 | 查找对象 |
|---|---|
| 技术栈 | `package.json`、`requirements.txt`、`pom.xml`、`build.gradle`、`vite.config.*`、`application.*` |
| 数据库 | `.sql`、`.db`、`models`、`entity`、`schema`、`migrations` |
| 后端功能 | `controller`、`router`、`service`、`api`、`views` |
| 前端功能 | `pages`、`views`、`components`、`router`、`store` |
| AI 功能 | `predict`、`model`、`train`、`ai`、`forecast`、`analysis` |
| 测试证据 | `tests`、`test`、截图、日志、数据库样例 |

项目专用验证清单可以写成 JSON，并交给 `scripts/verify_project_content_completeness.py` 校验。清单内容必须来自本项目源码，不能沿用其他论文的接口或实体。

## 5. 截图分析流程

系统功能截图不是装饰材料，而是界面和测试证据。处理截图时：

1. 先按功能分类，例如登录注册、任务管理、时间记录、效率统计、AI 预测、个人中心等。
2. 为每张截图记录页面名称、展示功能、可支撑章节。
3. 每张拟放入论文的截图前必须有正文介绍，图后可有解释。
4. 截图不能代替实现说明，实现章必须说明功能如何由代码、接口和数据支撑。
5. 测试章截图必须与测试用例表对应，不能只贴界面图。
6. 若截图信息与源码不一致，以源码和可运行结果为准，并在修改说明中记录。

## 6. 章节内容映射

建议按以下映射组织初稿：

| 章节 | 主要输入 | 必须写清楚 | 禁止事项 |
|---|---|---|---|
| 摘要 | 任务书、开题报告、源码结论 | 研究对象、主要功能、技术方法、实现结果 | 不引用参考文献，不写未实现功能 |
| 1 绪论 | 开题报告、参考文献 | 背景、意义、研究现状、论文结构 | 不与摘要雷同，不堆百科 |
| 2 需求分析 | 任务书、源码、截图 | 可行性、角色、功能需求、非功能需求 | 不写源码没有的角色和功能 |
| 3 系统设计 | 源码结构、数据库、图表参考 | 架构、模块、流程、数据库、AI 模型 | 图表与代码/表结构矛盾 |
| 4 系统实现 | 源码、截图 | 核心功能实现逻辑、关键接口、页面交互 | 只写操作步骤，不写实现 |
| 5 系统测试 | 截图、可运行结果、测试证据 | 测试环境、测试用例、结果分析 | 所有结果都写“通过”且无证据 |
| 6 结论与展望 | 全文事实 | 已完成工作、不足、改进方向 | 写个人感想或夸大成果 |
| 参考文献 | 正文引用、近年资料 | 顺序编码、格式完整 | 文献与正文引用不一致 |
| 致谢 | 用户资料或通用表达 | 简洁、得体 | 出现敏感隐私或过度口语 |

## 7. 图表生成决策

论文图表必须来自项目事实，不得为了凑数量凭空绘制。

| 图表类型 | 依据 | 放置章节 |
|---|---|---|
| 系统功能模块图 | 源码模块、任务书功能要求 | 需求分析或系统设计 |
| 系统架构图 | 前后端目录、接口、数据库、AI 模块 | 系统设计 |
| 业务流程图 | 路由、页面流程、控制器逻辑 | 需求分析或系统设计 |
| E-R 图 | 数据表和字段关系 | 数据库设计 |
| 数据表结构表 | 数据库文件、模型类、迁移脚本 | 数据库设计 |
| 核心界面图 | 系统功能截图 | 系统实现 |
| 测试用例表 | 功能需求和可验证结果 | 系统测试 |

图表样式必须参考 `论文各种图表的样式参考` 文件夹，并遵守主指南第 10、11、13 节。

结构图、流程图、模块图、部署图、接口调用图和 E-R 图必须统一为：

- 纯白背景。
- 黑色文字。
- 黑色线条和箭头。
- 不使用灰色填充、彩色边框、主题色、渐变、网格背景或黑底。
- 同一篇论文内生成图必须使用一致字体；优先采用宋体或与参考图等效的中文字体。
- 图中文字必须完整落在框内；生成后需用 PDF/PNG 截图检查。

总 E-R 图不能只画少数物理表。若系统源码中有总览、统计分析、AI 预测、风险评估、部署配置、演示数据初始化等业务主题，也应在总 E-R 图或相邻说明中覆盖，并明确哪些是逻辑主题、哪些是物理表。

总 E-R 图必须先做布局清单再出图：

1. 把实体框、逻辑主题框和关系菱形放到行列网格中，先保证同层水平关系和上下层垂直关系不交叉。
2. 每条关系分配独立连接端口，避免多个关系从同一框的同一点出线。
3. 关系菱形与两侧框之间保留可见短线，不能贴框；`1`、`n` 标记放在连接线附近空白处。
4. 输出 `er_layout.json`，至少包含 `boxes`、`relations[].diamond_box`、`relations[].segments`、`relations[].cardinality_boxes`。
5. 运行 `scripts/verify_er_layout_geometry.py er_layout.json`，通过后再插入 `.docx`；脚本通过后仍要看 PDF/PNG 截图，确认线条不乱、文字不压框。

代码图如果不是直接截取 IDE，也应生成成 IDE/编辑器截图风格：保留白色编辑区、行号或标题栏、等宽字体和适度语法高亮。若代码字符串中含中文，必须使用中文字体回退；生成脚本同时输出 `code_figures_manifest.json`，并运行 `scripts/verify_code_figure_text.py code_figures_manifest.json`，防止中文字符串在最终图片中变成 `????` 或缺字方框。

## 8. 内容生成顺序

推荐写作顺序如下：

1. 先完成证据台账。
2. 根据证据台账确定目录和章节标题。
3. 写需求分析、系统设计、系统实现、系统测试，全文按不少于 20000 个非空白可见字符组织；初稿建议留出 22000-25000 字余量。
4. 回头写摘要、绪论、结论。
5. 整理图表编号和正文引用。
6. 整理参考文献，保证正文引用顺序一致。
7. 写致谢。
8. 按主指南生成 Word 文档。
9. 按封面规范处理前两页。
10. 按目录规范和自动化脚本规范更新目录、固化 TOC 格式。
11. 导出 PDF，做最终验收。

这样可以避免摘要先写得很漂亮，但正文和项目事实对不上的问题。

## 9. 禁止编造清单

以下内容没有证据时不得写入论文：

- 未在源码、截图或任务书中出现的功能模块。
- 未在数据库、模型类或代码中出现的数据表和字段。
- 未能确认的第三方接口、算法模型、训练数据集。
- 未实际执行或没有截图/日志支撑的测试结果。
- 性能指标、准确率、响应时间、用户数量等具体数值。
- 与项目技术栈不一致的框架、语言和部署环境。
- 参考文献中不存在的观点或正文没有引用的文献。

如果论文需要这些内容，必须先补证据或明确写成“后续可扩展方向”。

## 10. 交付前验证

交付前必须至少完成以下验证：

1. 主指南 `.docx` 或生成论文 `.docx` 能通过 `officecli validate` 或等效 OpenXML 校验。
2. 前两页导出 PDF/PNG 后，与封面参考图结构一致。
3. 运行 `verify_cover_identifiers_and_length.py`，确认封面右上角学号/档号结构正确且全文非空白可见字符数不少于 20000。
4. 正文目录结构与章节实际顺序一致。
5. 自动目录经过 Word COM 更新，并在必要时经过 OpenXML 后处理固化 TOC 缩进、行距和制表位。
6. 图表编号、正文引用和图表位置一致。
7. 数据库表、功能模块、API、前端视图、AI 功能描述能回溯到源码或截图证据。
8. 测试用例表与测试截图或结果说明对应。
9. 参考文献数量和格式满足主指南要求。
10. 修改说明记录使用了哪些资料，以及哪些内容因证据不足未写入。
11. `verify_monochrome_diagrams.py` 对结构/流程/E-R 类图返回 `MONOCHROME_DIAGRAM_CHECK=PASS`。
12. `verify_project_content_completeness.py` 使用项目专用 JSON 返回 `PROJECT_CONTENT_COMPLETENESS=PASS`。

若任何验证失败，不得声称论文已完成，必须先修正或明确标注未完成项。

## 11. 给后续 agent 的最短执行摘要

后续 agent 接手时，先做以下动作：

1. 读 `AGENTS.md`，确认使用 Superpowers。
2. 读主格式指南，掌握全论文 Word 规则。
3. 读封面前两页规范，避免封面错版。
4. 读目录页规范，避免目录手写、点线错位和层级不足。
5. 读自动化脚本规范，理解 Word COM 更新字段和 OpenXML 固化格式的必要性。
6. 读本清单，按证据台账组织内容。
7. 分析任务书、开题报告、源码、截图和图表参考。
8. 先建立证据，再写正文。
9. 先生成内容，再套 Word 格式，最后导出 PDF 验收。
